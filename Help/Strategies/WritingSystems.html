OurWord Help
     Writing Systems

General

We have struggled with the best way to handle writing systems for decades now. I do not claim to have done any better than other software; in fact, it is highly probable that what I've done here in OurWord is quite inadequate. Which likely means a revision is in the future. But in the meantime.....here is what we have:

A writing system, loosely defined, is intended to be all of the information about displaying, sorting, and keyboarding for a given alphabet. A language might have multiple such systems; but in most cases a language has only one.

Some examples of what writing systems describe:

  • That in Spanish, "ll" should be sorted as a single symbol
  • That in Tagalog, "ng", "n" and "g" are all separate symbols.
  • That in many Chinese languages, a "SimSun" font will display its fonts correctly.

In LinguaLinks (and then in FieldWorks), a writing system was defined at the "language" level, meaning that a given language could have a number of writing systems.

In designing OurWord, I have had a goal of wanting the software to work for many languages with as little setup as possible. A part of that is that I am attempting to permit a Writing System to be defined apart from individual languages. Thus if all of the languages in a cluster can use a single Writing System, it is not necessary define it over and over again each time a new translation is begun.

Within the StyleSheet, for each individual style, you are able to define a different font (font name, size, etc.) for each writing system. Thus a Book Title in Chinese will likely have a different font than in English; and yet for the paragraph style, you only need to define "centered" once.

I expect eventually merge OurWord writing systems with the model that is supported by WeSay.

Configuration

The Configuration Dialog, under the Team Settings section, allows for writing systems information to be entered. Currently the following are allowed for each:

  • A name for the writing system, which is the way to identify it elsewhere in the software
  • A short abbreviation / ID, which is currently used in interfacing with the WeSay dictionary
  • A keyboard name, which is used to support automatic switching of keyboards (both Windows and KeyMan keyboards are supported.) Note: you need to enter the name of the keyboard, not the abbreviation. Thus, e.g., enter "Thai Kedmanee" rather than "TH."
  • Punctuation: a list of punctuation in the writing system
  • End Punctuation: a list of punctuation that can occur following a word.

Once a writing system is set up, then under a translation's settings (e.g., under the "Target" tab under "Essentials", you need to enter the writing system for the two kinds of text: Advisor and Vernacular.

  • Vernacular - the translation
  • Advisor - the back translation

Once this is done, keyboard switching should be operational.

Concerning general-purpose writing systems

It is normal in language software to set up a separate writing system for each language. However, if you note that the various languages in a cluster all use the same orthography, keyboardings, etc., it is permissable to set up a single writing system and have them all use it. OurWord defaults to a "Latin" writing system, which will work for the many languages that use an English orthography. Thus many languages can go ahead and use OurWord out of the box; others with more complicated languages will need to do additional setup.

Windows Repeated Keyboard Name Bug

Windows has an unfortunate idea of what a program wants when it asks it for the list of keyboards. Rather than give the actual name of the keyboard, it often gives the name of a keyboard elsewhere in the list. As a result, you might see the same keyboard name multiple times. This only happens when you have multiple keyboards under a single language; thus the workaround is to move each keyboard under its own separate language in your "Regional and Language Options" control panel. That is, you probably need to pretend that your keyboard is actually some other language. See http://wesay.org/wiki/Windows_Repeated_Keyboard_Name_Bug for more details.
